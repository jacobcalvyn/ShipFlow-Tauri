# Service Mode Architecture Design

## Status

This document defines the safest path to add an optional local or LAN API service to the existing desktop app.

It is a design document only. It does not imply that the feature is already implemented.

## Goals

- Keep the current desktop app as the primary product
- Add an optional API service mode inside the same app
- Allow other apps to call the tracking engine without creating a separate project
- Support background operation with:
  - Windows system tray
  - macOS menu bar / tray icon
- Require authentication for every non-health API endpoint
- Generate and rotate the auth token only from inside the app
- Keep the default mode safe:
  - service disabled
  - no public exposure

## Non-Goals

- Expose the desktop app directly to the public internet
- Replace the desktop UI with a headless-only service
- Add multi-user auth or role-based access control
- Add full API management or developer portal features
- Add cloud sync or remote secret storage in the first version

## Product Model

The app should have two runtime roles:

1. Desktop workspace UI
2. Optional local service host

These roles should live in the same app process but must remain logically separated:

- the desktop UI owns settings and operator workflows
- the service host owns HTTP traffic and auth
- the tray / menu bar owns service visibility and lifecycle shortcuts

## High-Level Architecture

```text
Tauri App
  -> Desktop UI
    -> Settings Dialog
    -> Workspace / Sheets
  -> Service Mode Controller
    -> HTTP Server
    -> Auth Middleware
    -> Request Logging
    -> Tracking Engine Adapter
  -> Tray / Menu Bar Controller
    -> Show App
    -> Start / Stop API
    -> Copy Endpoint
    -> Copy Token
    -> Quit
```

## Service Modes

Three modes are enough for the first version.

### 1. Disabled

Default mode.

- no API listener
- no background service behavior

### 2. Local API

Recommended mode.

- bind to `127.0.0.1`
- accessible only from the same machine
- intended for local companion apps, scripts, or browser tools

### 3. LAN API

Advanced mode.

- bind to `0.0.0.0`
- accessible from the local network
- must display explicit risk warning in settings
- must require auth token

The app should not expose an “internet public” mode.

## Config Model

The service configuration should be versioned and stored in app-local config.

```ts
type ServiceMode = "disabled" | "local" | "lan";

type ServiceConfig = {
  version: 1;
  enabled: boolean;
  mode: ServiceMode;
  port: number;
  authToken: string;
  launchOnStartup: boolean;
  startInBackground: boolean;
  minimizeToTray: boolean;
  lastUpdatedAt: string;
};
```

### Rules

- `enabled=false` always means no listener
- `mode="disabled"` is equivalent to `enabled=false`
- `port` should default to a fixed safe port, for example `18422`
- `authToken` must always exist when service mode is enabled
- `authToken` is generated by the app, never typed manually

## Settings UX

The existing `Setting` dialog should gain a new `API Service` section.

Suggested fields:

- `Enable API Service`
- `Service Mode`
  - `Disabled`
  - `Local API`
  - `LAN API`
- `Port`
- `Auth Token`
  - `Reveal`
  - `Hide`
  - `Copy`
  - `Generate`
  - `Regenerate`
- `Launch on Login`
- `Start in Background`
- `Minimize to Tray / Menu Bar`

### Apply Model

Settings should keep the current preview-confirm pattern:

- changing values previews in UI state
- `OK` persists config
- `Batal` rolls back to the previous saved config

For token regeneration:

- `Regenerate` should mark a pending new token in preview state
- `OK` confirms and persists it
- `Batal` discards it

This avoids invalidating live clients accidentally before the user confirms.

## Auth Token Design

### Generation

Token should be generated only inside the app using secure randomness.

Recommended:

- 32 random bytes
- encoded as URL-safe base64 or hex

Example:

```text
sf_4C7o...<redacted>
```

### Rules

- no manual editing field
- `Generate` only when missing
- `Regenerate` requires explicit confirm
- old token becomes invalid immediately after new token is persisted
- token should be hidden by default in the UI

### Storage

First version can store the token in local app config.

Better follow-up version:

- use OS-native secure storage if available

But config storage is acceptable for the initial version if:

- file stays local to the user profile
- token is never logged

## API Contract

First version should stay minimal.

### Health

```http
GET /health
```

Response:

```json
{ "ok": true }
```

No auth required.

### Status

```http
GET /status
Authorization: Bearer <token>
```

Response:

```json
{
  "service": "running",
  "mode": "local",
  "bindAddress": "127.0.0.1",
  "port": 18422
}
```

### Track by Shipment ID

```http
GET /track/:shipment_id
Authorization: Bearer <token>
```

Response should reuse the existing `TrackResponse` shape already returned by `track_shipment`.

### Optional POST Variant

```http
POST /track
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "shipmentId": "P2603310114291"
}
```

This is optional but useful for future batching or structured input.

## Auth Middleware

All endpoints except `/health` should require:

```http
Authorization: Bearer <token>
```

Rules:

- missing token -> `401`
- malformed header -> `401`
- wrong token -> `401`
- token should be compared without leaking details in response text

No token should appear in:

- logs
- error strings
- tray labels
- copied endpoint strings

## Tray / Menu Bar Behavior

When service mode is enabled, the app should provide tray/menu bar controls.

Suggested menu items:

- `Show App`
- `Start API`
- `Stop API`
- `Copy Local Endpoint`
- `Copy Auth Token`
- `Open Settings`
- `Quit`

Tray title / tooltip can show compact status:

- `API Off`
- `API Local :18422`
- `API LAN :18422`

## Window Lifecycle Rules

### When service mode is disabled

- close window can quit normally

### When service mode is enabled

- close window should hide to tray / menu bar
- app should remain alive in background
- full quit should happen only from:
  - tray / menu bar `Quit`
  - explicit app quit action

### Startup

If `launchOnStartup=true` and `startInBackground=true`:

- app starts hidden
- service starts automatically
- tray / menu bar becomes the primary entrypoint

## Service Lifecycle

The app should centralize service state in one runtime controller.

```ts
type ServiceRuntimeState = {
  status: "stopped" | "starting" | "running" | "stopping" | "error";
  mode: ServiceMode;
  bindAddress: string | null;
  port: number | null;
  errorMessage: string | null;
};
```

### Rules

- only one active listener at a time
- settings apply should trigger controlled restart if bind mode or port changed
- failed start should keep the app alive and show a clear error
- tray status should reflect runtime state, not saved config alone

## Binding Rules

### Local API

- bind `127.0.0.1:<port>`

### LAN API

- bind `0.0.0.0:<port>`

When mode changes:

- stop previous listener
- start new listener
- update tray status

## Error Handling

Important cases:

- port already in use
- invalid port
- failed listener bind
- failed restart after config change
- token missing while enabling service

User-facing behavior:

- keep settings dialog open on validation errors
- show precise error message
- do not half-apply service state

## Logging

Service mode should add lightweight logs:

- service start
- service stop
- bind address and port
- auth failures
- request path
- request result
- request duration

Do not log:

- auth token
- full sensitive headers

## Security Rules

### Required

- service off by default
- token required for every non-health endpoint
- LAN mode must show a warning before confirm
- no public internet mode in UI

### Strongly Recommended

- lightweight rate limit for LAN mode
- request timeout
- request size limits for POST endpoints
- origin / CORS kept closed unless explicitly needed

## Implementation Plan

### Phase 1

- add `ServiceConfig`
- add secure token generation / regeneration
- expose config in `Setting` dialog

### Phase 2

- add HTTP server controller in Rust
- add `/health`, `/status`, and `/track/:shipment_id`
- add auth middleware

### Phase 3

- add tray / menu bar integration
- hide-to-tray behavior when service mode is enabled

### Phase 4

- add LAN mode warning UX
- add request logging and light rate limiting

## Recommended Defaults

```ts
const DEFAULT_SERVICE_CONFIG: ServiceConfig = {
  version: 1,
  enabled: false,
  mode: "disabled",
  port: 18422,
  authToken: "",
  launchOnStartup: false,
  startInBackground: false,
  minimizeToTray: true,
  lastUpdatedAt: new Date().toISOString(),
};
```

## Final Recommendation

The safest product path is:

- keep this as one app
- add optional service mode
- default to disabled
- recommend local-only mode
- require token auth
- use tray / menu bar for background lifecycle

This gives the user a reusable local API without splitting the project into a separate server application.
